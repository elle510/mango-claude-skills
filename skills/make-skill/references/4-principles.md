# 스킬 작성 4대 원칙

## 원칙 1: 뻔한 말은 쓰지 마라

Claude가 이미 아는 것은 스킬에 넣지 않는다.

**스킬에 넣지 말아야 할 것:**

- API 호출 방법, fetch 사용법 등 범용 개발 지식
- TypeScript 타입 작성법, React 훅 사용법
- Git 기본 명령어

**스킬에 반드시 넣어야 할 것:**

- 이 팀/프로젝트만 아는 규칙 (예: "커밋 메시지는 한국어로")
- 이 코드베이스의 함정 (예: "index.ts 없이 내부 경로 직접 import 금지")
- Claude의 기본 성향을 깨는 지침 (예: "보라색 그라데이션 쓰지 마", "요약 대신 트리거 조건으로 써")

**자기 검증 질문:** "이 내용을 Claude에게 말하지 않아도 알까?" → YES면 지운다.

## 원칙 2: Gotchas 섹션이 가장 중요하다

Gotchas = Claude가 반복적으로 실수하는 지점의 오답 노트.

**작성 방법:**

1. 처음엔 짧아도 된다. 두세 줄로 시작해도 좋다.
2. Claude가 스킬을 실행하다 실수할 때마다 그 케이스를 추가한다.
3. 선배가 신입에게 팁을 주는 형식으로 쓴다.

**좋은 Gotcha 형식:**

```
- **[실수 유형]**: 구체적인 설명. → 올바른 방법.
```

**예시:**

```
- **description 요약 실수**: "이 스킬은 배포를 한다"처럼 요약 쓰지 말 것.
  → "사용자가 프로덕션에 배포하려 할 때"처럼 트리거 조건으로 써야 한다.
- **기존 스킬 덮어쓰기**: 수정 전 반드시 기존 내용을 Read로 읽고 시작한다.
  → 읽지 않으면 Write 도구가 실패한다.
```

**Gotchas가 없는 스킬은 미완성이다.** 처음엔 "아직 없음"이라도 섹션을 만들어 둔다.

## 원칙 3: 점진적 공개 (Progressive Disclosure)

모든 내용을 SKILL.md 한 파일에 넣으면 컨텍스트 낭비다.

**규칙:**

- SKILL.md = 개요 + 절차 요약 + 파일 목록 포인터
- 상세 내용 = `references/`, `assets/` 분리 파일
- Claude가 필요할 때만 해당 파일을 읽도록 설계

**분리 기준:**
| SKILL.md에 둘 것 | 분리할 것 |
|-----------------|----------|
| 절차 요약 (5단계 이내) | API 문서, 파라미터 설명 |
| 파일 포인터 ("자세한 건 X.md 참고") | 긴 예시 코드 |
| 핵심 Gotchas | 템플릿, 보일러플레이트 |

**SKILL.md가 100줄 넘어가면** 분리를 고려한다.

## 원칙 4: Description은 요약이 아니라 트리거 조건이다

settings.json의 description 필드는 Claude가 "이 스킬을 언제 써야 하는가"를 판단하는 신호다.

**나쁜 예 (요약):**

```
"이 스킬은 배포 관련 작업을 도와준다."
"FSD 아키텍처 리뷰 스킬."
"커밋 메시지를 자동으로 생성한다."
```

**좋은 예 (트리거 조건):**

```
"사용자가 서비스를 프로덕션에 배포하려 하거나, PR 머지 후 배포 상태를 확인하려 할 때 사용한다."
"사용자가 FSD 규칙 확인, 아키텍처 리뷰, import 방향 검사를 요청할 때 사용한다."
"사용자가 /commit을 입력하거나 변경사항을 커밋하고 싶다고 할 때 사용한다."
```

**공식:**

```
"사용자가 [구체적인 상황/행동]을 할 때 사용한다. [관련 키워드]를 포함 시 발동."
```
